# Project Content Questionnaire

> **Instructions for the Claude agent answering this:** You have access to this project's actual codebase, git history, README, and commit log — use them. Prefer real specifics (actual dates from `git log`, actual dependencies from `package.json`, actual feature names from the code) over generic phrasing. Where something genuinely isn't determinable from the repo (e.g. real-world impact metrics, the user's specific role/motivation), say so explicitly rather than inventing a plausible-sounding answer — flag it as "needs input from Shruti" instead of guessing.
>
> Answer every question below in order, keeping the numbering so Shruti can paste your answers straight back for transcription.

## Basic info
1. **Date range** — when did you actively work on this project? (Month Year -- Month Year, or "ongoing")
2. **One-line description** — a single sentence describing what this project is and who it's for.
3. **Tags** — 2-4 short labels for this project (e.g. tech stack, platform, or category — "Mobile App", "Figma", "React"). If nothing meaningful comes to mind, say "none" and we'll drop the tags field for this project.

## Context & scope (for the project detail page)
4. **Context** — a short paragraph: what the project is, who the client/users are (if any), and Shruti's specific role. Pull from README/commit messages if there's a stated purpose.
5. **Scope** — one line: what was actually built/delivered (e.g. "End-to-end mobile app design + front-end build").
6. **Duration** — how long the project took, in a resume-friendly format (e.g. "3 months").
7. **Role** — Shruti's specific role/title on this project (e.g. "Product Designer & Developer").
8. **Tools** — every tool actually used. Note: the site's schema currently only supports `figma` and `github` as tool icons — if other tools were core to this project (Framer, VS Code, specific libraries, etc.), list them anyway and flag that the schema may need extending.

## Key outcomes (exactly 3 needed)
9. Three short stat-style outcomes, each as a number/metric + 2-3 word label (e.g. "40% faster checkout", "600+ users", "3x conversion"). If no real metrics exist, say so — we'll need to decide whether to estimate, omit, or reframe as qualitative outcomes instead.

## Process (exactly 3 steps needed)
For each of 3 problems tackled during the project, answer:
10. **Problem statement** — what was the specific challenge (short phrase, e.g. "Problem 1: Onboarding drop-off").
11. **Description** — 1-2 sentences: research/options considered, decision made, obstacles navigated.

(Repeat for Problem 2 and Problem 3.)

## Final design
12. Is there a final design image/screenshot available to export? (Just yes/no — the actual image export is a separate manual step.)

## Learnings (exactly 3 needed)
13. Three specific things Shruti learned from this project — technical, process, or design lessons. Avoid generic statements ("I learned to communicate better") in favor of specific ones tied to this project.

## Resume-specific (in addition to the above)
14. If you had to compress this project into 1-2 resume bullet points (outcome-framed, not task-framed), what would they say?
